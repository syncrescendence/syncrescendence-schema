# Typed Contract Formats for Multi-Agent AI Commission: Schema Comparison

## Executive Summary

Six production-grade systems for commissioning work between agents were examined: OpenAI Agents SDK (Turn/Item/Thread primitives), Google A2A Protocol (v0.3), IBM's Agent Communication Protocol (ACP), CrewAI's task assignment schema, LangGraph's state-passing conventions, and OpenAI Symphony's WORKFLOW.md contract. Each encodes different ontological assumptions about what a "commission" is — ranging from a message-sent-to-a-Task-tracked-entity (A2A) to a typed state document passed across graph nodes (LangGraph) to a YAML-governed prompt template dispatched by a daemon scheduler (Symphony). The minimal required fields for a machine-parseable, auditable research commission synthesized from across all six systems are: `commission_id`, `issuer`, `assignee`, `description`, `expected_output`, `status`, `created_at`, and `trace_id`. Typed `input_schema` / `output_schema` and a `context` pointer are strongly implied by production practice, even where not formally required.

***

## 1. OpenAI Agents SDK: Turn, Item, Thread Primitives

### 1.1 Thread and Turn

OpenAI's Agents SDK (experimental Codex extension) structures agent execution around three nested primitives: **Thread**, **Turn**, and **Item**.[^1][^2]

- A **Thread** is a persistent session context identified by a `thread_id`, created via `thread/start` and reused across continuation turns. The `thread/start` call requires `approvalPolicy`, `sandbox` (SandboxMode), and `cwd` (absolute workspace path). Optional: declared tool specs for dynamic client-side tools.[^3][^4]
- A **Turn** is one round of agent execution within a thread, initiated by `turn/start`. Required fields: `threadId`, `input` (array of typed items), `cwd`, `title`, `approvalPolicy`, `sandboxPolicy`.[^5][^3]
- A **Thread/Turn Session ID** is derived as `<thread_id>-<turn_id>` and is the canonical observability identifier for a live agent session.[^5]

```json
// turn/start params (Symphony/Codex app-server protocol)
{
  "threadId": "<thread-id>",
  "input": [{"type": "text", "text": "<rendered prompt>"}],
  "cwd": "/abs/workspace",
  "title": "ABC-123: Example Issue",
  "approvalPolicy": "never",
  "sandboxPolicy": {"type": "workspaceWrite"}
}
```

The `TurnOptions` class adds per-turn config: `idle_timeout_seconds` and a cancellation `signal`. `ThreadOptions` sets session-level defaults: `model`, `model_reasoning_effort`, `network_access_enabled`, `web_search_mode`, `approval_policy`.[^2][^1]

### 1.2 RunItem Types

A `RunResult` accumulates items generated during a run in the `new_items` list. The `RunItem` type alias is a discriminated union:[^6][^7][^8]

| Item Type | Purpose |
|-----------|---------|
| `MessageOutputItem` | Final message output from the LLM[^9] |
| `ToolCallItem` | A tool invocation request generated by the LLM[^6] |
| `ToolCallOutputItem` | The result of a tool call[^6] |
| `HandoffCallItem` | Triggers agent-to-agent handoff; contains raw `ResponseFunctionToolCall`[^6] |
| `HandoffOutputItem` | Confirms handoff completion[^6] |
| `ReasoningItem` | Model reasoning step (chain-of-thought)[^6][^10] |
| `MCPListToolsItem` | MCP tool discovery event[^6] |
| `MCPApprovalRequestItem` | Tool approval gate[^6] |
| `ToolApprovalItem` | Tool execution approval pause (for interruption/HITL)[^7] |
| `CompactionItem` | History compaction marker[^6] |

A `RunResult` exposes: `input`, `new_items`, `raw_responses`, `final_output`, `input_guardrail_results`, `output_guardrail_results`, `tool_input_guardrail_results`, `last_agent`, `last_response_id`. The `last_response_id` is the lightweight continuation primitive for chaining turns across runs.[^11][^7][^12][^8]

### 1.3 RunConfig: The Commission's Control Envelope

`RunConfig` governs a full run and is the closest analogue to a typed commission header in the OpenAI SDK. Key fields:[^11]

- `workflow_name`, `trace_id`, `group_id`, `trace_metadata` — observability / audit identity[^11]
- `input_guardrails`, `output_guardrails` — constitutional constraint hooks[^11]
- `handoff_input_filter` — post-handoff context shaping[^11]
- `call_model_input_filter` — pre-model-call state editing[^11]
- `reasoning_item_id_policy` — controls whether reasoning IDs are preserved or stripped in continuation input[^11]
- `tracing_disabled`, `trace_include_sensitive_data` — audit policy toggles[^11]

**Confirmed fact**: Thread/Turn/Item is a concrete JSON-RPC-like protocol over stdio, with the normative contract in the Codex app-server spec. The OpenAI SDK `Runner` layer does not expose a single "commission" document type — commission semantics are assembled from `RunConfig` + initial `input` + `session`.[^13][^3][^5][^11]

## 2. Google A2A Protocol (v0.3): Task Object

### 2.1 Data Model

The A2A Protocol uses JSON-RPC 2.0 as its transport and defines a three-layer architecture: Canonical Data Model → Abstract Operations → Protocol Bindings (JSON-RPC, gRPC, HTTP/REST). The normative source of truth is `spec/a2a.proto`; all SDKs and JSON Schemas are derived from it.[^14][^15]

The **Task** is "the fundamental unit of work managed by A2A, identified by a unique ID". Its canonical fields from the A2A spec and Google Cloud REST API:[^16][^17][^14]

```json
{
  "id": "string",           // REQUIRED — unique task identifier
  "contextId": "string",    // optional — groups related tasks/sessions
  "status": {               // REQUIRED
    "state": "submitted|working|input-required|completed|canceled|failed|rejected|auth-required",
    "message": { ...Message },
    "timestamp": "RFC3339 string"
  },
  "artifacts": [{ ...Artifact }],   // optional — agent outputs
  "history": [{ ...Message }],      // optional — message history
  "metadata": { ...object },        // optional — freeform KV pairs
  "kind": "task"                    // type discriminator
}
```

The **Message** object (used both in `Task.history` and `SendMessageRequest`) carries:[^17][^16]

```json
{
  "messageId": "string",   // required
  "contextId": "string",   // optional
  "taskId": "string",      // optional — binds message to a task
  "role": "user|agent",    // required
  "content": [{ ...Part }],// required
  "metadata": {},
  "extensions": ["uri", ...]
}
```

A **Part** is the atomic content unit: `text | file | data` (discriminated union)[^16].

### 2.2 Task Negotiation via SendMessageRequest

Work is commissioned via `message/send`. The `SendMessageRequest` fields:[^18][^14][^17]

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `message` | `Message` | **Yes** | Content + metadata of commission |
| `configuration` | `SendMessageConfiguration` | No | Execution mode, push notification config, historyLength |
| `metadata` | `Record<string, any>` | No | Additional context |

`SendMessageConfiguration` controls:[^14]
- `returnImmediately` (boolean) — blocking vs. non-blocking execution
- `acceptedOutputModes` — MIME types the client accepts
- `historyLength` — conversation context window
- `taskPushNotificationConfig` — async webhook for long-running tasks

### 2.3 TaskState Lifecycle

Standard A2A states: `submitted → working → completed | failed | canceled | rejected | input-required | auth-required`[^19][^20]. The Bindu extension adds negotiation states: `negotiation-bid-submitted`, `negotiation-bid-lost`, `negotiation-bid-won`, plus `payment-required` and `trust-verification-required`[^19].

**Confirmed**: A2A v0.3 is the current published spec as of early 2026. The `kind: "task"` discriminator field was added to disambiguate Task from Message in streaming responses. **Contradiction to note**: Some community documentation shows an older `sessionId` field in the Task object, while the current normative spec uses `contextId` — the migration from `sessionId` to `contextId` appears to have occurred between early v0.2 and v0.3.[^19][^15][^20][^16][^14]

## 3. Agent Communication Protocol (ACP): Run Object

### 3.1 Provenance and Status

ACP was developed by IBM Research (open-sourced as `i-am-bee/acp`). The repository was archived in August 2025, suggesting the project was absorbed or superseded. Current ACP SDK documentation (via `agentcommunicationprotocol.dev`) continues to be maintained, indicating a fork or continuation — this is an **unconfirmed inference**.[^21][^22][^23][^24][^25]

### 3.2 Run Schema (OpenAPI 3.1, v0.2.0)

The **Run** is ACP's work unit, created via `POST /runs`. From the normative OpenAPI spec:[^25]

```yaml
Run:
  required: [agent_name, run_id, status, output, created_at]
  properties:
    agent_name:    string (RFC1123 DNS label, 1–63 chars)
    session_id:    UUID (optional — for stateful multi-turn)
    run_id:        UUID (required)
    status:        created|in-progress|awaiting|cancelling|cancelled|completed|failed
    await_request: AwaitRequest (nullable — HITL pause payload)
    output:        array of Message (required, may be empty)
    error:         Error (nullable)
    created_at:    datetime (required)
    finished_at:   datetime (optional)
```

The `RunCreateRequest` fields:[^25]

```yaml
RunCreateRequest:
  required: [agent_name, input]
  properties:
    agent_name:  string
    session_id:  UUID (optional)
    session:     Session object (optional, inline state)
    input:       array of Message (minItems: 1)
    mode:        sync|async|stream
```

### 3.3 Message and MessagePart

The ACP `Message` schema requires `role` and `parts`:[^21][^25]

```python
class Message(BaseModel):
    role: str  # "user" | "agent" | "agent/{name}"
    parts: list[MessagePart]
    created_at: datetime  # optional
    completed_at: datetime  # optional

class MessagePart(BaseModel):
    name: Optional[str]           # makes part an Artifact if set
    content_type: str             # REQUIRED — MIME type
    content: Optional[str]        # inline content OR
    content_url: Optional[AnyUrl] # URL reference (exclusive with content)
    content_encoding: Literal["plain", "base64"] = "plain"
    metadata: Optional[CitationMetadata | TrajectoryMetadata]
```

The `TrajectoryMetadata` type is notable: it carries audit trail information — `message` (reasoning step), `tool_name`, `tool_input`, `tool_output`. This makes ACP's MessagePart the only schema examined that has **native first-class support for tool execution auditing** embedded in the message structure itself.[^25]

### 3.4 Agent Manifest (Discovery Contract)

The `AgentManifest` is the capability declaration that precedes commission:[^25]

| Field | Required | Purpose |
|-------|----------|---------|
| `name` | Yes | RFC1123 slug identifier |
| `description` | Yes | Human-readable capability description |
| `input_content_types` | Yes | Accepted MIME types |
| `output_content_types` | Yes | Producible MIME types |
| `capabilities` | No | Structured list of `{name, description}` skill objects |
| `metadata.framework` | No | Agent framework (BeeAI, CrewAI, AutoGen, etc.) |
| `metadata.recommended_models` | No | Preferred model identifiers |

## 4. CrewAI Task Assignment Schema

### 4.1 Task Attributes

CrewAI's `Task` is a Pydantic-backed Python class representing a discrete assignment to a specific agent. Unlike A2A or ACP, there is no wire protocol — the schema exists as in-process Python object state.[^26][^27][^28]

| Attribute | Parameter | Type | Required | Description |
|-----------|-----------|------|----------|-------------|
| Description | `description` | `str` | **Yes** | What the task entails[^26][^28] |
| Expected Output | `expected_output` | `str` | **Yes** | Defines completion criteria and output format[^26][^28] |
| Agent | `agent` | `Optional[BaseAgent]` | **Yes** (in practice) | Responsible execution agent[^28] |
| Name | `name` | `Optional[str]` | No | Human identifier |
| Tools | `tools` | `List[BaseTool]` | No | Tool access scope for this task |
| Context | `context` | `Optional[List[Task]]` | No | Other tasks whose outputs feed as context[^26] |
| Output JSON | `output_json` | `Optional[Type[BaseModel]]` | No | Pydantic model for structured JSON output |
| Output Pydantic | `output_pydantic` | `Optional[Type[BaseModel]]` | No | Pydantic model for typed output |
| Output File | `output_file` | `Optional[str]` | No | Path to persist output |
| Async Execution | `async_execution` | `Optional[bool]` | No | Parallel execution flag |
| Human Input | `human_input` | `Optional[bool]` | No | HITL review gate before finalization |
| Guardrail | `guardrail` | `Optional[Callable]` | No | Output validator (retry on failure) |
| Guardrail Max Retries | `guardrail_max_retries` | `Optional[int]` | No | Default: 3[^29][^26] |
| Callback | `callback` | `Optional[Any]` | No | Post-completion hook |
| Markdown | `markdown` | `Optional[bool]` | No | Force Markdown output format |
| Config | `config` | `Optional[Dict[str, Any]]` | No | Task-specific freeform config |

**Confirmed minimum required**: `description` + `expected_output` + `agent`. YAML-based task definition is the recommended authoring format.[^28][^26]

### 4.2 Execution Modes

Crew processes are either `Process.sequential` (chain) or `Process.hierarchical` (role-based routing). The `context` field implements a **data dependency graph** — a task can declare other tasks as input providers, with LangGraph-style DAG semantics but within a single process boundary.[^29][^26]

## 5. LangGraph: State-Passing Conventions

### 5.1 StateGraph and StateSchema

LangGraph does not have a single "task commission" object — work is represented as **typed state documents** passed between nodes in a compiled graph. The `StateGraph` constructor signature:[^30][^31][^32][^33]

```python
StateGraph(
    state_schema: type[StateT],       # REQUIRED — TypedDict, Pydantic, or dataclass
    context_schema: type[ContextT] | None = None,  # immutable runtime context
    input_schema: type[InputT] | None = None,       # input subset
    output_schema: type[OutputT] | None = None      # output subset
)
```

Each state field can be annotated with a **reducer** controlling how parallel node updates are merged:[^32][^34]

```python
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]  # append reducer
    task_id: str
    status: str
    result: Optional[str]
```

### 5.2 Private vs. Public State

LangGraph supports **private state channels** — a node can accept a narrower `Input` schema than the full `OverallState`, and can write to any channel in the full graph state. This enables commission sub-contracts: a subgraph or node only exposes the fields relevant to its capability, hiding internal working state.[^31][^35][^30]

In subgraphs with **different state schemas**, the parent graph must include a translation node that maps subgraph outputs back into the parent's state.[^35]

### 5.3 Thread and Checkpointing

LangGraph's concept of a "thread" is the `thread_id` in `RunConfig.configurable`. This is the persistence key for checkpointing — it associates a conversation/run with a durable state store, enabling `interrupt()` + `Command(resume=...)` for human-in-the-loop workflows.[^36][^37][^38][^39]

The `Interrupt` dataclass carries `value` (any JSON-serializable payload to surface to the human) and `id` (deterministic hash used as resume key).[^39]

### 5.4 Commission Semantics in LangGraph

There is no native "commission receipt" type. By convention, production LangGraph systems encode commission metadata as fields in the `OverallState` TypedDict. The `group_id` in tracing config links related traces (analogous to `context_id` in A2A). LangGraph's typed state boundaries are the enforcement mechanism for schema contracts — violations raise `ValidationError` at schema check time, not at runtime propagation.[^11]

## 6. Symphony WORKFLOW.md: Repository-Owned Commission Contract

### 6.1 Architecture

OpenAI's Symphony (open-source, `openai/symphony`) is a daemon scheduler that continuously polls a Linear issue tracker, creates per-issue workspaces, and dispatches Codex coding agents. The commission contract lives entirely in `WORKFLOW.md` — a Markdown file with YAML front matter that is version-controlled in the target repository.[^3][^5]

The design principle is that the workflow file "should be self-contained enough to describe and run different workflows (prompt, runtime settings, hooks, and tracker selection/config) without requiring out-of-band service-specific configuration".[^5]

### 6.2 WORKFLOW.md Front Matter Schema

Top-level keys:[^5]

| Key | Required | Description |
|-----|----------|-------------|
| `tracker.kind` | Yes | `"linear"` (only supported tracker) |
| `tracker.api_key` | Yes | Literal token or `$VAR_NAME` |
| `tracker.project_slug` | Yes | Linear project identifier |
| `tracker.active_states` | No | Default: `["Todo", "In Progress"]` |
| `tracker.terminal_states` | No | Default: `["Done", "Closed", ...]` |
| `polling.interval_ms` | No | Default: 30000 (supports hot reload) |
| `workspace.root` | No | Per-issue workspace root path |
| `hooks.after_create` | No | Shell script on workspace creation |
| `hooks.before_run` | No | Shell script before each agent attempt |
| `hooks.after_run` | No | Shell script after attempt (logged, not fatal) |
| `agent.max_concurrent_agents` | No | Default: 10 |
| `agent.max_turns` | No | Default: 20 |
| `agent.max_concurrent_agents_by_state` | No | Per-state concurrency caps |
| `codex.command` | Yes | Agent executable (e.g., `codex app-server`) |
| `codex.turn_timeout_ms` | No | Default: 3600000 (1 hour) |
| `codex.stall_timeout_ms` | No | Default: 300000 (5 min); 0 disables stall detection |

### 6.3 Issue as Commission Object

The normalized `Issue` entity is the commission carrier in Symphony's domain model:[^5]

```
Issue:
  id, identifier, title, description, priority, state,
  branch_name, url, labels, blocked_by, created_at, updated_at
```

The `RunAttempt` is the execution receipt:[^5]

```
RunAttempt:
  issue_id, issue_identifier, attempt (null=first, int=retry),
  workspace_path, started_at, status, error
```

The `LiveSession` tracks real-time observability:[^5]
- `session_id` = `<thread_id>-<turn_id>`
- `turn_count`, `codex_input_tokens`, `codex_output_tokens`, `codex_total_tokens`
- `last_codex_event`, `last_codex_timestamp`, `last_codex_message`

### 6.4 Prompt Template Contract

The Markdown body of `WORKFLOW.md` is a Liquid-compatible template. Strict rendering is required — unknown variables or filters are errors, not silent empty strings. Available template variables: `issue` (full normalized issue object) and `attempt` (integer or null).[^5]

**Confirmed**: Symphony dispatches continuation turns by sending only continuation guidance to the existing thread rather than resending the original prompt — making the full issue context a one-time commission payload, with subsequent turns as incremental guidance.[^3][^5]

## 8. Minimal Required Fields for a Machine-Parseable, Auditable Research Commission

Synthesizing across all six systems, the following fields constitute the **minimal viable typed contract** for issuing work between agents with full auditability. Fields are grouped by function:

### 8.1 Identity Layer (mandatory)

```json
{
  "commission_id": "uuid",           // immutable, globally unique
  "issuer_id": "string",             // issuing agent/system identifier
  "assignee_id": "string",           // receiving agent/system identifier
  "created_at": "ISO8601 datetime",  // immutable creation timestamp
  "schema_version": "semver string"  // contract version for parsing compatibility
}
```

**Rationale**: All six systems encode unique identity. ACP uses UUID `run_id`; A2A uses string `id`; Symphony uses `issue.id`. `schema_version` is derived from production practice — without it, contract parsers cannot handle schema evolution safely.[^42][^14][^25][^5]

### 8.2 Work Description Layer (mandatory)

```json
{
  "description": "string",            // what to do (CrewAI: description; A2A: message.content)
  "expected_output": "string",        // completion criteria and output format (CrewAI)
  "input_schema": "JSON Schema | null", // typed input contract
  "output_schema": "JSON Schema | null" // typed output contract
}
```

**Rationale**: `description` + `expected_output` are CrewAI's only required fields. A2A encodes both as message content. ACP encodes them as MIME-typed `MessagePart` arrays. LangGraph enforces them as TypedDict annotations. Production typed-schema practice strongly recommends `additionalProperties: false` for strict validation.[^43][^28][^17][^32][^42][^25]

### 8.3 Routing and Context Layer (recommended)

```json
{
  "context_id": "string | null",       // session/thread grouping (A2A: contextId; ACP: session_id)
  "context_refs": ["commission_id"],    // upstream commissions whose outputs are inputs (CrewAI: context)
  "priority": "integer | null",         // dispatch priority (Symphony: issue.priority)
  "labels": ["string"]                  // capability routing tags
}
```

### 8.4 Lifecycle and Status Layer (mandatory)

```json
{
  "status": "pending|active|waiting|completed|failed|cancelled",
  "status_updated_at": "ISO8601 datetime",
  "error": "string | null",
  "attempt": "integer | null"   // retry counter (Symphony: RunAttempt.attempt)
}
```

### 8.5 Observability and Audit Layer (mandatory for constitutional governance)

```json
{
  "trace_id": "string",              // distributed trace identifier (OpenAI SDK: trace_id)
  "receipt": {
    "started_at": "datetime | null",
    "finished_at": "datetime | null",
    "token_usage": {"input": 0, "output": 0, "total": 0},  // Symphony pattern
    "tool_calls": [                  // ACP TrajectoryMetadata pattern
      {
        "tool_name": "string",
        "input": {},
        "output": {},
        "timestamp": "datetime"
      }
    ]
  },
  "artifacts": [                     // A2A Artifact pattern
    {
      "artifact_id": "string",
      "name": "string",
      "content_type": "MIME type",
      "content": "string | null",
      "content_url": "URI | null"
    }
  ]
}
```

**Rationale**: The `trace_id` enables cross-system correlation. `receipt` with token counts enables cost attribution and replay auditability (Symphony operational model). `tool_calls` within receipt is derived from ACP's `TrajectoryMetadata` — the only production schema examined that natively embeds tool execution provenance in the message layer. `artifacts` following A2A's `artifactId`/`name`/`parts` pattern enables structured output delivery.[^17][^25][^5][^11]

### 8.6 Trust and Governance Layer (constitutional systems)

```json
{
  "constitutional_refs": ["rule_id"],  // grammar rules governing this commission
  "approval_policy": "never|auto|human_required",  // Symphony/Codex pattern
  "guardrails": ["guardrail_id"],      // CrewAI guardrail / OpenAI SDK guardrail
  "human_in_loop": false,              // CrewAI: human_input; LangGraph: interrupt()
  "sandbox_mode": "none|read|workspace-write"  // Thread-level isolation (Codex)
}
```

## 9. Cross-System Tensions and Contradictions

### 9.1 Commission vs. Message Ambiguity (A2A)

A2A explicitly debates when to use a `Task` versus a `Message` for a response. The spec allows agents to return a direct `Message` for simple interactions without creating a Task — meaning a commission response can arrive without a persistent trackable entity. This undermines auditability for simple exchanges: if the agent returns `Message` instead of `Task`, there is no `id` to query later.[^44][^14]

### 9.2 Schema Enforcement Boundaries

LangGraph enforces schema contracts at **graph compile time** via TypedDict/Pydantic type annotations, but node-to-node state passing is typed only at the Python process boundary — a node can write any valid key to state at runtime. A2A enforces schema at the HTTP boundary via JSON Schema but the normative source is a `.proto` file. CrewAI enforces output contracts via `guardrail` callbacks, not at the wire layer. Only ACP has native MIME-typed content negotiation built into the protocol structure.[^15][^33][^26][^31][^14][^25]

### 9.3 Identity and Observability Gaps

CrewAI has no native commission ID, timestamp, or token accounting — all observability is delegated to callbacks or external logging. LangGraph's `thread_id` is a conversation session identifier, not a per-task commission ID — a single thread can contain multiple logical commissions without distinguishing them. OpenAI SDK's `trace_id` and `group_id` are **optional** and not enforced — commission tracing requires developer opt-in.[^27][^38][^26][^39][^11]

### 9.4 Synchrony Assumptions

A2A is designed async-first; `returnImmediately: false` (default) is blocking but the system is architected for long-running tasks with push notifications and polling[^14]. ACP offers explicit `mode: sync|async|stream` selection per run[^25]. Symphony's scheduler is inherently async — the orchestrator runs a poll loop and dispatches workers, with continuation retries after each worker exits[^5]. LangGraph's `interrupt()` is a **synchronous pause** within an async execution model — the graph pauses at the checkpoint, requires a `Command(resume=...)` to proceed, and depends on a durable checkpointer[^36][^39]. These incompatible synchrony models mean cross-framework commission chaining requires explicit mode negotiation.

### 9.5 Discovery Layer

A2A's `AgentCard` is a published capability manifest that clients must fetch before commissioning work — it declares skills, endpoint, auth schemes, and supported output modes. ACP's `AgentManifest` (via `GET /agents/{name}`) serves the same function with richer metadata including `capabilities`, `domains`, `recommended_models`, and `dependencies`. CrewAI, LangGraph, and OpenAI SDK have no equivalent network-level discovery — agent capability is known at code-authoring time, not at runtime negotiation. Symphony's discovery is issue-tracker-driven — capability matching is implicit in the prompt template, not declared in a machine-readable capability schema.[^33][^26][^13][^14][^17][^25][^5]

***

## 10. Recommendations for a Constitutional Grammar Layer

For a multi-agent system entering "operational constitution," the following schema design decisions are actionable:

1. **Adopt A2A's Task object as the base commission envelope** but augment it with CrewAI's `expected_output` as a required field. A2A provides the strongest wire-level identity (`id`, `contextId`, `status` lifecycle, `artifacts`) and the widest adoption surface (Google, Bindu, and community implementations).[^19][^15][^14]

2. **Embed ACP's `TrajectoryMetadata` pattern inside artifacts or receipts** to provide per-tool-call audit trails within the commission body itself, rather than as external log infrastructure. This is the only production schema examined with first-class tool provenance in the message layer.[^25]

3. **Use LangGraph's `Annotated[type, reducer]` pattern for typed state boundaries** between agents with different capability scopes — private state channels prevent state bleed across trust boundaries.[^30][^31][^33]

4. **Adopt Symphony's `WORKFLOW.md` pattern for the constitutional grammar layer** itself — a version-controlled, hot-reloadable policy document with strict template rendering (unknown variables fail hard) that governs runtime behavior without out-of-band configuration. The `max_concurrent_agents_by_state` map is directly applicable for trust-tiered dispatch.[^3][^5]

5. **Treat `trace_id` + `group_id` + `commission_id` as a three-level hierarchy**: `commission_id` identifies the atomic work unit; `group_id` links all commissions in a research workflow; `trace_id` is the distributed tracing span identifier for cross-system debugging. This maps to A2A's `id` / `contextId` / A2A-Trace-Header pattern.[^14][^11]

6. **Enforce `output_schema` at the contract level, not just the agent level.** Production typed-schema practice uses `additionalProperties: false` JSON Schema with `status: "unknown"` as the fallback for insufficient data, bounded retries (2 max), and escalation on schema violation. Validation should happen at every agent boundary, not only at final output.[^43][^42]
