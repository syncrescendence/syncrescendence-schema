## I. Vanguard's verdict

The orchestration pattern is substantively sound. The eleven-avatar taxonomy holds. The skill bifurcation is the right abstraction. The relay chain, artifact-centric output, salon protocol, and thread lifecycle rules represent a real advance from substrate toward operations. The remaining risk is not architectural incoherence — it is concentration of ambiguity at three specific seams, plus the recurring deferral pattern.

Vanguard's sharpest diagnosis: the project has moved from "no traffic law" to "traffic law exists, but several intersections have ambiguous signage." That is a much better problem, and the corrective is not redesign but tightening.

## II. Corrections to adopt

### Correction 1 — Commander/Cowork routing law

Vanguard's formulation is crisp enough to adopt verbatim as doctrine:

**Commander** is the constitutional executive for governed production. Default executor for any task that mutates repo state, evaluates code, depends on MCP/Git/Chrome chains, or touches the singular present-state artifact. Its native objects are the working tree, the CLI, Git state, and deliberate multi-step production.

**Cowork** is the operations concierge for governed intake, recurrence, and light transformation. Default executor for recurring operational support, connector-mediated intake, app ecosystem workflows, scheduled surveillance, and pre-repo staging work. Its native objects are connectors, plugins, scheduled tasks, and web-to-document pipelines.

The system fails not because they overlap but if both are allowed to be generalists at the same decision altitude. The routing test: does the task require filesystem truth, branch discipline, deterministic execution, or constitutional write sensitivity? → Commander. Does it require web traversal, connector aggregation, scheduled recurrence, or app-mediated transformation? → Cowork.

### Correction 2 — The three-Projects vertical separation

Claude.ai Projects, Cowork Projects, and repo-level CLAUDE.md are not parallel containers. They sit at different layers:

- **Claude.ai Project** — salon-layer deliberative memory and synthesis context.
- **Cowork Project** — operations-layer persistent workspace context.
- **CLAUDE.md** — repo-layer execution law.

This vertical separation must be encoded explicitly in all custom instructions. "Project" is a false friend that collapses three distinct scopes into one mental bucket unless the doctrine repeatedly clarifies the distinction.

### Correction 3 — Oracle mode-tagging

Oracle remains one avatar, two instruments. The correction is not avatar fission but receipt differentiation. All Oracle outputs must carry a mode tag: `[oracle:grok.com-sensing]` or `[oracle:api-decomposition]`. This prevents cross-mode confidence laundering — live X traversal feels epistemically alive but is structurally weak; grammar-injected decomposition feels sterile but is institutionally cleaner. The system must always know which evidentiary profile produced a given claim.

### Correction 4 — Augur special-operations as provisional compression

"Special operations" is workable as a first-order distinction but semantically coarse. Computer actually covers three use classes: high-friction research, cross-app intervention, and genuinely exceptional missions. The governing check for Computer invocation sharpens to: invoke Computer only when its distinctive powers are required — parallel integrated search plus tool-bearing action plus persistent task memory plus integration reach. If that criterion is absent, the task belongs to Commander, Cowork, or a simpler research surface. "Cannot reach" must not silently expand into "did not want to route carefully."

### Correction 5 — Commission equalization for skill-less surfaces

The bifurcation is correct but commission quality must be equalized. The mechanism: move shared doctrinal overhead out of the commission body and into stable reusable artifacts — a compressed roster block, canonical state pointers, a fixed relay contract, and the artifact template. These are pasted consistently into every commission. The commission itself carries only task-specific deltas plus minimum doctrine for lawful execution. Skill-less surfaces receive commissions heavier in transport but not noisier in cognition. The commission is a stable operating shell, not ad hoc overhead.

The four failure modes to design against: doctrine drift (inline packet diverges from skill-carried doctrine), compression drift (heavy CI becomes lossy under character pressure), behavioral asymmetry (richer tasks go to skill-capable agents while skill-less agents get bloated one-offs), and authority bleed from pseudo-skills (ChatGPT GPTs or Grok agents become de facto doctrine stores outside the repo).

Pseudo-skill workarounds (GPTs, Grok agents) are acceptable only as frozen packaging layers for already-ratified doctrine, never as primary doctrine containers.

### Correction 6 — Staged validation, not monolithic

Build order shifts from "build all → validate at end" to staged doctrinal hardening with per-phase validation. Each phase builds a pair, immediately tests a live relay, and only proceeds after proving commission legibility.

**Phase A — Core deliberative pair + validation.**
Build Claude.ai Project CI (lean, skill-capable) and ChatGPT Project CI (heavy, skill-less) simultaneously. Apply Corrections 1–6. Immediately validate: one commission from Vizier to Vanguard, relay back, synthesis pass. Check for doctrine leakage, routing ambiguity, and artifact-format compliance.

**Phase B — Research surface + validation.**
Build Grok Projects CI (heavy, skill-less). Apply Oracle mode-tagging correction. Immediately validate: one adversarial research commission to Oracle, relay back, synthesis pass. Check source attribution, mode tag presence, and evidence profile clarity.

**Phase C — Execution surface review + validation.**
Review repo-level CLAUDE.md for Commander. Add orchestration pattern awareness, Chrome integration, Commander/Cowork routing law. Immediately validate: one Commander execution task (commit an artifact to repo, update HANDOFF-CURRENT.md). Check governed write path and state-pointer resolution.

With the core quartet (Vizier, Vanguard, Oracle, Commander) instruction-stable, run the calibration cycle. Target: Slack. **Phase E — Post-cycle buildout.**
Extract lessons from cycle receipts. Build remaining 10 actions (Perplexity, Cowork, Augur template, remaining CLI configs) informed by operational evidence rather than theory. The cycle tells us what the skills and CIs actually need to carry.

## IV. Decisions for Sovereign ratification

1. **Accept the orchestration pattern as substantively sound.** No structural redesign needed.
2. **Adopt the Commander/Cowork routing law** (Correction 1). Encode in both CIs and in CLAUDE.md.
3. **Adopt the three-Projects vertical separation** (Correction 2). Encode in all five project CIs.
4. **Adopt Oracle mode-tagging** (Correction 3). `[oracle:grok.com-sensing]` / `[oracle:api-decomposition]` on all Oracle outputs.
5. **Treat Augur special-operations as provisional compression** (Correction 4). Sharpen invocation threshold.
6. **Adopt commission equalization** (Correction 5). Build a reusable doctrinal packet (roster block, state pointers, relay contract) that is pasted consistently into all commissions regardless of surface class.
7. **Adopt staged validation** (Correction 6). Per-phase, not monolithic.
8. **Cut CI build scope to minimum viable set** (5 actions for cycle readiness, not 15). Defer 10 actions to post-cycle.
9. **Run the calibration cycle with partial infrastructure.** Frame as bounded proof-of-traffic, not full-fleet validation.
