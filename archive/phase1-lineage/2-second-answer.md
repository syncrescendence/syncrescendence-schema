## I. The Research Primitives That Produced harness-taxonomy.md

The lineage reveals a three-stage research architecture, each stage with its own recurring primitives. What made harness-taxonomy.md possible was not any single move but the disciplined composition of all three stages.

### Stage 1: Per-Entity Deep Decomposition (the directive-report pairs)

Seven directives were dispatched to Grok's council, one per harness. Each directive is itself a structured research instrument with recurring formal elements:

**Source stratum specification.** Every directive prescribes exactly four source tiers, always in the same priority order: first-party documentation and source code; core developer primary authority (named individuals, specific X handles, PR histories); community consensus distilled to repeatable patterns (named subreddits, Discord servers, GitHub discussions); bleeding-edge pioneers (named forks, experimental repos, undocumented workflows). This is not "do research." It is a four-layer epistemic stack where each layer contributes a distinct evidential character — official mechanics from tier 1, behavioral nuance from tier 2, empirical convergence from tier 3, frontier signal from tier 4. The CI provenance report calls this "triangulation" but the actual structure is quadrangulation with explicit tier semantics.

**Pre-specified decomposition axes.** Each directive names the exact structural dimensions to decompose: configuration precedence hierarchy, root and dot-file surface, memory architecture, self-improvement loops, core execution loops, multi-agent orchestration, patterns and antipatterns, prompting formalism, extensibility surface. These axes were held constant across all seven harnesses — the same skeleton applied to Aider as to OpenClaw as to OpenHands. This is the critical move that made cross-harness synthesis possible downstream. Without isomorphic decomposition axes, the reports would have been incommensurable — you cannot extract unified primitives from structurally incompatible descriptions.

**Progressive depth stacking.** Each harness got three to four sequential prompts, each explicitly building on the prior response as immutable context. The pattern is consistent: core architecture → orthogonal power-user spectra → absolute frontier spectra → culmination synthesis. Each subsequent prompt expands the decomposition surface while referencing the prior response by section number to prevent restating. This is a depth-first research strategy: exhaust one layer before opening the next, accumulating context window mass that forces the model into increasingly specific territory. The directives explicitly prohibit preamble, restatement, and overview — every sentence must deepen.

**Output grammar constraints.** Every directive specifies the same output discipline: mechanism first, then concrete example, then risk/reversibility. Epistemic status must be marked inline. Code blocks for exact configs. Tables for schemas. No summaries, no hedging. This is not style preference — it is an output grammar that ensures the reports contain extractable primitives rather than narrative gloss. The phrase "dense technical prose: mechanisms first, conclusions derived, zero preamble" recurs verbatim across directives.

**Falsifiability injection.** Directives require each claim to carry a falsifiable test or reproduction command. The Aider directive asks for "measurable mastery tiers and real-user velocity metrics." The Codex directive asks for "inline answers to incisive probes" with specific edge cases named. The OpenHands directive demands "reproduction commands/files" for each mechanism. This turns the reports into testable specifications rather than plausible narratives.

**Council architecture as built-in adversarial check.** Grok's four-agent council (Harper for evidence grounding, Benjamin for logical stress-testing, Lucas for divergent synthesis) provides structural triangulation within each response. The directives explicitly invoke this: "council tensions: e.g., Claude reliability vs local Qwen3 cost." This means each report has already been internally stress-tested before it arrives at the synthesis stage.

### Stage 2: Cross-Entity Synthesis (how the taxonomy was compiled from the reports)

The taxonomy itself was produced by a separate synthesis operation — likely a session where all seven reports were loaded as context and decomposed into unified versus idiosyncratic primitives. The synthesis primitives are visible in the taxonomy's structure:

**Isomorphic alignment.** Because all seven reports shared the same decomposition axes, the synthesizer could align them column by column: "what does Aider call its config cascade? What does Claude Code call the same structural concept? What does Codex call it?" The shared skeleton made this alignment mechanical rather than interpretive.

**Convergence extraction.** A primitive was promoted to "unified" (Section 1) if it appeared in five or more of six harnesses (OpenCode was apparently culled during synthesis). The threshold is implicit but consistent — the taxonomy flags which harnesses instantiate each primitive and where the weakest instance is. This is a quorum rule: a primitive earns universal status through empirical convergence, not definitional decree.

**Divergence preservation.** Primitives that appeared in fewer than five harnesses, or that carried significantly different semantics across harnesses, were preserved in per-harness idiosyncratic sections (Section 2 onward). The taxonomy does not force false unity — it holds the divergence as structurally informative. Aider's PageRank repo map, Claude Code's managed enterprise tier, OpenClaw's ClawWork economic model — these stay idiosyncratic because collapsing them would destroy information.

**Cluster formation.** The taxonomy organizes unified primitives into named clusters (A through S) by functional domain rather than by harness. This is a rotation of the analysis frame: the reports are organized by entity (one harness at a time), the taxonomy is organized by function (one capability domain at a time). The rotation is what makes the taxonomy feel "most decomposed" — it separates what things do from who does them.

**Naming normalization.** The taxonomy assigns canonical names to primitives that appear under different labels in different harnesses. "Persistent Instruction Files" unifies CLAUDE.md, AGENTS.md, GEMINI.md, CONVENTIONS.md, and SKILL.md under a single structural concept. This is the normalization move that transforms a pile of per-harness terminology into a shared vocabulary.

### Stage 3: Compression Without Loss (the output grammar of the taxonomy itself)

The taxonomy achieves its density through a specific compression strategy visible in every entry:

Each primitive gets a **bolded definitional name** (the canonical term), a **one-to-two-sentence mechanism description** (what it is and how it works structurally), and a **cross-platform instantiation list** (which harnesses implement it, with what naming, and where the strongest/weakest instances are). No entry exceeds a short paragraph. No entry contains narrative context about why the primitive matters — the structural position within the cluster carries that weight. This is aggressive compression that preserves governing structure: you can reconstruct the full landscape from the taxonomy's entries because each entry carries both its definition and its cross-platform mapping.

## II. The Generalizable Research Packet

Those primitives compose into a reusable inquiry schema. Here is the packet, abstracted from the specific harness domain:

### The Inquiry Schema for Producing Taxonomy-Grade Artifacts

**Phase 1: Entity-Level Deep Decomposition**

For each entity in the target domain (each SaaS surface, each ontology layer, each config subsystem, each control-plane component):

*Decomposition axes* — define a fixed skeleton of structural dimensions that will be held constant across all entities. The axes must be chosen to be isomorphic across the domain: they must be applicable to every entity, even if a given entity's answer to a given axis is "this dimension does not apply" or "this dimension is trivially simple here." For the harness domain, the axes were: config precedence, root/dot-files, memory, self-improvement, execution loops, multi-agent, patterns/antipatterns, prompting, extensibility. For the exocortex domain, the axes would be something like: primitive object model, manipulation path (API/CLI/MCP/webhook/browser/manual), data sovereignty (where truth lives, who can write), auth and trust boundary, receipt and audit surface, teleological role and anti-role, integration priority, failure modes, reversibility, epistemic status.

*Source strata* — define four tiers of evidence, always in priority order. Tier 1: first-party documentation from the surface's developer (official docs, API references, changelogs). Tier 2: primary authority from the surface's core developers or architects (named individuals, specific public statements). Tier 3: community consensus from heavy users (forums, community repos, operational war stories). Tier 4: bleeding-edge practitioners (unusual integrations, undocumented behaviors, experimental workflows). Each tier contributes different epistemic character. Not all tiers will be available for every entity — and that absence is itself informative about the entity's maturity.

*Progressive depth* — dispatch three to four sequential prompts per entity, each building on the prior as immutable context: core architecture → operational spectra → frontier/edge cases → culmination synthesis. Each prompt must explicitly prohibit restatement and require reference to prior sections by number.

*Output grammar* — enforce: mechanism first, then example, then risk/reversibility, then epistemic status marker. Dense prose, no preamble, no narrative framing, no conclusions. Code blocks for exact configs. Tables for typed structures.

**Phase 2: Cross-Entity Synthesis**

Load all entity-level reports into a single context. Apply:

*Isomorphic alignment* — align reports by their shared decomposition axes. For each axis, extract what each entity does.

*Convergence extraction with quorum rule* — a primitive earns "unified" status if it appears in a threshold proportion of entities (e.g., 4/6, 5/7). Below threshold, it remains entity-specific.

*Divergence preservation* — preserve entity-specific primitives in dedicated per-entity sections. Do not force false unity.

*Cluster formation by function* — rotate the organization from per-entity to per-function. Group unified primitives by what they do, not who does them.

*Naming normalization* — assign a single canonical term to each unified primitive. Collapse synonyms explicitly. Note the per-entity terminology as aliases.

**Phase 3: Compression**

Apply the taxonomy output grammar: bold definitional name, one-to-two-sentence mechanism description, cross-entity instantiation map. No entry exceeds a short paragraph. Position within the cluster hierarchy carries contextual weight — entries do not need to explain why they matter.

The correct treatment: ARCHITECTURE.md is canon for the repo-native coding harness layer. It is evidence (strong evidence, well-structured evidence) for the broader system architecture. It is not the seed from which the broader architecture should be grown — the broader architecture requires its own research pipeline at each layer, using the same method that made ARCHITECTURE.md credible in the first place.
