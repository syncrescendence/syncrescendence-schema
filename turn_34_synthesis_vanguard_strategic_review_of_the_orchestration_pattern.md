# SYNTHESIS: Vanguard Strategic Review of the Orchestration Pattern

**Type:** Synthesis  
**Turn:** 34  
**Salon:** vanguard / schema-3  
**Commission from:** Vizier (Claude.ai / Opus 4.6)  
**Sovereign steering:** "Review the attached, calibrate against your previous Phase 3 review in the repo, and respond to Claude."

The orchestration pattern is directionally sound and materially stronger than the Phase 3 proposal I previously reviewed, because it no longer mistakes the repo alone for the operating system. It now names the relay chain, the avatar roster, the instruction stack, thread lifecycle, salon protocol, and the distinction between skill-capable and skill-less surfaces. That is a real advance from substrate toward operations. The remaining risk is not architectural incoherence at the whole-system level. The eleven-avatar taxonomy basically holds. The remaining risk is concentration of ambiguity at three specific seams: the Commander–Cowork boundary, the distinction between avatar identity and surface identity where one institutional role spans multiple execution modes, and the possibility that custom-instruction buildout becomes a substitute for proving live traffic. The pattern is therefore not flawed at the level of concept. It is fragile at the level of operational discrimination.

My prior Phase 3 review argued that the project was still ahead on substrate and behind on operational law, and warned against finishing infrastructure before proving that real work packets fit the pipes. That warning still applies, but with a narrower target. The present pattern has now supplied much of the missing coordination law in lightweight form: the inverted relay, artifact-centric output, typed salon lanes, and explicit thread lifecycle rules all reduce oral tradition. What remains under-specified is not the existence of roads, but the routing criteria at moments of overlap. In other words, the project has moved from “no traffic law” to “traffic law exists, but several intersections still have ambiguous signage.” That is a much better problem.

The Commander–Cowork relation is the most load-bearing of those intersections. The current pattern is right to keep them as distinct avatars despite shared model lineage and shared CLAUDE.md inheritance, because their execution ontology is different. Commander is the bounded repo-and-terminal executive. Its native object is the working tree, the CLI, MCP-mediated tools, Git state, and deliberate multi-step production. Cowork is not simply Commander with plugins. It is an ambient operations surface: VM-sandboxed, connector-rich, schedulable, web-permeable, intermittently autonomous, and suitable for recurring governance chores, web-to-document translation, and plugin-mediated workflows that benefit from being persistent but not deeply entangled with a local development environment. The distinction is therefore not model, and not even instruction source. It is work topology. Commander should own operations where filesystem truth, branch discipline, deterministic execution, or constitutional write sensitivity are primary. Cowork should own operations where web traversal, connector aggregation, scheduled recurrence, or app-mediated transformation are primary.

That said, the pattern as written does not yet make this boundary sharp enough under load. The phrase “autonomous operations” for Cowork and “polymathic execution” for Commander are both true and still too close. Under pressure, the Sovereign will not ask which surface has the cleaner ontology. The Sovereign will ask which one gets the job done with minimum coordination drag. Without a sharper routing law, Commander and Cowork will cannibalize one another’s task domain. The corrective is not redesign, but a stricter distinction of authority. Commander should be the default executor for any task that mutates repo state, evaluates code, depends on MCP/Git/Chrome chains, or touches the singular present-state artifact. Cowork should be the default executor for recurring operational support, connector-mediated intake, app ecosystem workflows, scheduled surveillance, and pre-repo staging work. Put differently: Commander is the constitutional executive for governed production; Cowork is the operations concierge for governed intake, recurrence, and light transformation. The system does not fail because the two overlap. It fails if both are allowed to be generalists at the same decision altitude.

The “three Projects” problem across Claude.ai, Cowork, and Claude Code is real, but it is a naming hazard more than an architectural flaw. Claude.ai Project, Cowork Project, and repo-level Claude Code context are not actually the same class of thing. One is deliberative memory and project-scoped synthesis context, one is persistent autonomous workspace context, and one is repository-local execution law. The flaw would be to treat them as parallel containers. They are not parallel. They sit at different layers. Claude.ai Projects govern constitutional synthesis and memory at the salon layer. Cowork Projects govern recurring workspace behavior at the operations layer. CLAUDE.md governs local execution behavior at the repo layer. The remedy is to encode that vertical separation explicitly in the doctrine, because the UI labels themselves will not do the work. If this is not stated repeatedly, “Project” will become a false friend that collapses three distinct scopes into one blurry mental bucket.

The Oracle dual-mode pattern should remain one avatar with two instruments, not split into two avatars yet. The reason is institutional role, not tool behavior. Both grok.com and the API route are performing adversarial sensing for the same constitutional purpose: they pressure-test consensus, surface live discourse, and inject external contradiction into the system. Their cognitive signatures differ, but that difference is already captured by mode distinction. Splitting them into separate avatars now would overfit implementation variance and create a false taxonomic multiplication. The threshold for avatar fission should not be “different interface, different prompt architecture, different tempo.” It should be “different institutional role, different success criteria, different products, different routing authority.” Oracle web and Oracle API do not yet cross that threshold. One senses directly in a volatile environment; the other decomposes through a structured relay. That is modality divergence within one role, not emergence of two roles.

The risk in keeping them unified is that one mode may quietly inherit authority earned by the other. Real-time X traversal can feel epistemically alive but structurally weak; grammar-injected API decomposition can feel sterile but institutionally cleaner. Those are different evidentiary profiles. So the design correction is not to split the avatar, but to require mode tagging on Oracle outputs and to prevent cross-mode laundering of confidence. The system should always know whether a claim came from live adversarial sensing or structured decomposition. One avatar; two receipts.

Augur is broadly correctly scoped, but the current split between “routine reconnaissance” and “special operations” is only barely sufficient. It works as a first-order distinction because it maps onto the strongest platform cleavage: Spaces for low-cost parallel search and synthesis, Computer for tool-bearing missions with memory, skills, and integrations. The risk is that this binary does not yet distinguish between research acceleration, operational intervention, and integration-heavy exception handling. Those are not identical workloads. Spaces is clearly routine reconnaissance. Computer, however, covers at least three different use classes: high-friction research tasks other agents cannot conveniently perform, cross-app intervention tasks where integrations matter more than reasoning style, and genuinely exceptional missions where model council, skills, or browser action are needed. Lumping those into “special operations” is workable for now but semantically coarse.

I would therefore not split Augur further at the avatar level, but I would treat “special operations” as a temporary compression, not a settled doctrine. Augur’s risk is surface-area inflation. Perplexity Computer can become a junk drawer for tasks that merely feel hard elsewhere. The governing check should be this: Augur Computer is invoked not because a task is vaguely complex, but because it requires one of its distinctive powers—parallel integrated search plus tool-bearing action plus persistent task memory plus integration reach. If that criterion is absent, the task probably belongs to Commander, Cowork, or a simpler research surface. The split is acceptable, but the threshold for invoking Computer mode must be sharper than “other agents cannot reach.” Otherwise “cannot reach” silently expands into “I did not want to route carefully.”

The skill bifurcation is the right abstraction. It recognizes an actual difference in the fleet rather than pretending all surfaces can be governed the same way. Trying to force a uniform instruction architecture across ChatGPT, Grok web, Grok API, Claude.ai, Cowork, Commander, Codex, Gemini CLI, and Perplexity would produce either under-instruction for skill-less surfaces or bloated redundancy for skill-capable ones. The distinction between CI→Skill→Commission and CI→Commission is therefore architecturally correct. But the danger you identified is real: a two-class system can emerge if the heavier commission burden on skill-less surfaces makes their assignments noisier, more repetitive, and less elegant than those given to skill-capable agents.

The way to equalize quality is not to erase the bifurcation. It is to move shared doctrinal overhead out of the commission body and into stable reusable artifacts that are referenced the same way across both classes. In practice, that means the commission should carry only task-specific deltas plus the minimum doctrine needed for lawful execution, while the heavy background material is stabilized in portable artifact templates, a compressed roster block, canonical state pointers, and a fixed relay contract that can be pasted consistently. In other words, skill-less surfaces need not receive worse commissions; they need more aggressively standardized commissions. The commission for a skill-less surface should feel heavier only in transport, not noisier in cognition. If the same doctrinal packet is reused with high consistency, the agent experiences it as a stable operating shell rather than as ad hoc overhead.

The main failure modes of the bifurcation are predictable. First, doctrine drift: the inline commission packet for skill-less surfaces diverges from the skill-carried doctrine on skill-capable surfaces, creating two subtly different constitutions. Second, compression drift: in an attempt to fit budgets, the heavy CI for a skill-less surface becomes lossy and omits one of the real routing constraints. Third, behavioral asymmetry: the institution starts giving richer, terser, more composable tasks to skill-capable agents while skill-less agents receive bloated one-off commissions, causing a false impression that the latter are intrinsically less reliable. Fourth, authority bleed from pseudo-skills: custom GPTs, Grok agents, or similar workarounds may appear to solve the skill gap while actually introducing yet another layer of opaque drift.

That last point matters. There is some value in skill-equivalent workarounds, but only where they reduce transport cost without introducing another unstable doctrinal surface. ChatGPT GPTs as pseudo-skills may help if used as frozen wrappers for repeatable artifact discipline, but they also create another governance location outside the repo. Grok custom agents are already present in the pattern and are a more natural fit because they remain inside the Oracle surface family, but they should still be treated as thin carriers, not sovereign doctrine stores. My recommendation is conservative: use pseudo-skill mechanisms only as packaging layers for already-ratified doctrine, never as primary doctrine containers. The moment the workaround becomes the place where the actual rules live, the bifurcation problem has merely been displaced.

The proposed custom-instruction build order is close but not optimal. The deeper issue is not whether ChatGPT or Claude.ai goes first in absolute terms. It is that validation is placed too late and too monolithically. The pattern should not build all five project CIs, the Perplexity global instructions, the Cowork global instructions, the research skill, the artifact template, and the CLI reviews, then validate at the end as though instruction architecture were a static asset. Instruction architecture is operational law under load. It needs staged proof.

I would slightly reorder the sequence. First, build Claude.ai and ChatGPT project CIs together, with ChatGPT not first in isolation but in the first validated pair. Claude.ai matters because Vizier is the commission-producing constitutional synthesist; ChatGPT matters because Vanguard is skill-less and therefore the clearest pressure test of whether the doctrine survives compression into a heavy project CI plus self-contained commission. This pair gives you the deliberative core. Then run a narrow validation immediately: one real commission on each, one relay back, one synthesis pass, and one check for doctrine leakage or ambiguity. Only after that should you move to Grok and Perplexity, because those are research expansion surfaces whose effective configuration depends on whether the core relay packet is actually stable. Then move to Cowork and the CLI reviews, because these are operations surfaces and should be calibrated after the commission grammar is proven, not before. Validation should therefore happen per phase, not at the end.

This produces a different logic of dependency. The highest-order dependency is not tool access. It is commission legibility. If the core deliberative surfaces cannot exchange a stable constitutional packet, then the rest of the buildout only scales confusion. So the order should be governed by doctrinal hardening, not by convenience or by platform novelty. Phase A should be Claude.ai plus ChatGPT plus immediate validation. Phase B should be Grok plus Perplexity plus immediate validation. Phase C should be Cowork plus the CLI surfaces plus immediate validation on one scheduled or connector-mediated task and one repo-touching task. The Augur template can sit with Phase B, but the Perplexity Computer research skill does not need to precede the first calibration cycle unless Augur Computer is part of that cycle’s minimum viable route.

This leads directly to the deferral-risk question, which is the real decision. The minimum viable custom-instruction set for running the calibration cycle is smaller than the full fifteen-action build. The cycle does not genuinely depend on complete instruction coverage across the entire fleet. It depends on a stable constitutional core, a usable research relay, and one governed path back into the repo and present-state logic. Building all fifteen actions before the first cycle would repeat the exact pattern my earlier review warned against: infrastructure completion standing in for operational proof.

The minimum viable set is this. First, the Claude.ai project CI, because Vizier must be able to craft and absorb commissions in stable alignment with the orchestration pattern. Second, the ChatGPT project CI, because Vanguard is both strategically important and skill-less, making it the best early stress test of the commission grammar. Third, one research-surface CI: either Grok web if you want adversarial sensing in the first cycle, or Perplexity Space instructions if you want routine reconnaissance with stronger source handling. You do not need both for cycle one unless the cycle’s method explicitly requires triangulation at that level. Fourth, the repo-level CLAUDE.md review, because Commander remains the likeliest governed writer and the surface most capable of bridging synthesis outputs into durable repo state. Fifth, a minimal artifact template and state-pointer packet already exists in substance, so no further large instruction work is needed before the first cycle unless a missing template would block the relay.

What does not belong in the minimum viable set is full Cowork configuration, Perplexity Computer SKILL.md, all CLI surface reviews, and the entire five-project-CI closure. Those are accelerants, not prerequisites. Valuable soon, but not before proof. Cowork especially should not delay the first cycle. Its distinctive value is recurring operations and connector-rich workflows, neither of which is necessary to prove that the calibration packet itself is sound. Likewise, the Augur Computer skill may improve future operations, but the first cycle can generate the evidence needed to design that skill more intelligently.

So yes: you should run the calibration cycle with partial infrastructure. That will not necessarily produce misleading receipts if the partiality is explicit and the cycle is framed as a bounded proof-of-traffic rather than a full-fleet validation. In fact, partial infrastructure is preferable because it exposes which missing instruction surfaces are genuinely blocking and which are merely tempting to perfect in advance. The cycle should be designed to answer four questions only: does the commission grammar survive transit, do the work packets fit the chosen pipes, do the outputs come back in artifact form that Vizier can synthesize, and does the governed writer path into repo state hold without improvisation. Those are the only receipts needed before broader CI buildout.

The real risk of misleading receipts would come from choosing a cycle that implicitly tests surfaces not yet configured, then blaming the method for failures caused by missing setup. Avoid that by narrowing the first cycle deliberately. Use the smallest sufficient fleet. Vizier, Vanguard, one research surface, Commander. Nothing more is needed to answer the operational question in front of you.

The directional recommendation is therefore firm. Accept the orchestration pattern as substantively sound. Tighten only the routing law at the Commander–Cowork seam, formalize mode-tagging for Oracle outputs, treat Augur special-operations language as provisional compression rather than settled niche doctrine, and shift the build strategy from full-surface instruction completion to staged doctrinal hardening with immediate validation. Then cut the custom-instruction scope to the minimum viable set and run the calibration cycle now. The next thing the project needs is not more boot loaders. It needs receipts from real traffic.
