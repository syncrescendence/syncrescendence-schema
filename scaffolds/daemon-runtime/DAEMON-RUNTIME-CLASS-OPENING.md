## 1. Modality Class Definition

**Source:** LAYER-LAW.md § 4; PRE-BUILD-MEMO-4-RUNTIME-MODALITY.md.

**Defining primitives of the daemon/runtime class:** Persistent autonomous operation across sessions; self-reporting operational status; task dispatch and receipt emission; variable autonomy levels; identity and accountability frameworks; potentially economic self-sustainability.

**Distinction from adjacent classes:** Daemon/runtime surfaces operate primarily on local machine state (files, repos, local services) with persistent identity. This distinguishes them from coding harnesses (session-bounded, repo-centric, code-authorship-focused), reasoning/chat surfaces (stateless between sessions, conversation-bounded), and autonomous operator surfaces like Manus (browser-mediated, remote-UI-focused).

**Named entities:** OpenClaw (fully autonomous daemon), Claude Cowork (supervised persistent agent).

**Internal differentiation:** The class contains a significant autonomy spectrum. OpenClaw is designed for unsupervised autonomous operation including economic self-sustainability. Cowork is designed for supervised task dispatch with human oversight at each significant action. These are different operating regimes within the same modality class — comparable to Aider vs OpenHands within coding harnesses.

## 2. Proposed Decomposition Axes

The daemon/runtime class requires its own axis set, structurally distinct from both SaaS (9 axes) and reasoning/chat (7 axes). The following axes are proposed based on the class's defining primitives and the evidence available. This axis set is provisional — it may be refined when scaffold production begins.

1. **Persistence and session architecture** — How the surface maintains state across sessions: session-spanning memory, persistent files, heartbeat/status documents, economic ledger.

2. **Autonomy model** — The degree and kind of autonomous operation: supervised task dispatch (Cowork) vs unsupervised campaign execution (OpenClaw). What decisions the surface can make without human approval. What veto/checkpoint mechanisms exist.

3. **Dispatch and task architecture** — How tasks are received, queued, decomposed, and executed. Single-task vs multi-task vs campaign-level operation. Parallel task execution. Queue management.

4. **Execution environment** — Where and how the surface executes: local filesystem sandbox (Cowork's Apple Virtualization Framework VM), local process (OpenClaw's terminal), cloud infrastructure, hardware embodiment (OpenClaw IoT pipelines).

5. **Identity and accountability** — How the surface maintains identity: SOUL.md (OpenClaw's liability/ethics file), institutional role assignment, operator relationship, audit trail generation, and accountability for actions taken.

6. **Economic model** — How the surface relates to costs: operator-funded (Cowork's subscription model), self-funding (OpenClaw's ClawWork), cost-awareness, credit exhaustion mitigations.

7. **Self-reporting and observability** — How the surface communicates its status: HEARTBEAT.md (OpenClaw), task progress display (Cowork), health indicators, planning horizon documents.

8. **Trust boundary and sandboxing** — What the surface can access and what it cannot: folder-scoped sandboxing (Cowork), workspace-level scoping (OpenClaw), credential management, prompt-injection defenses.

9. **Extensibility and connector architecture** — How the surface extends its capabilities: MCP connectors (Cowork), skills/Packs (OpenClaw), browser automation (Cowork + Claude in Chrome), hardware bindings (OpenClaw).

## 4. Per-Surface Research Packet: Claude Cowork

### Evidence Inventory

**Moderate evidence (recent, multiple sources):**
- Anthropic's official announcement (2026-01-12): Claude Cowork as "Claude Code for the rest of your work"
- VentureBeat coverage (2026-01-13): detailed feature description, genesis from Claude Code non-coding usage, Apple Virtualization Framework discovery
- DataCamp tutorial (2026-01-16): hands-on feature documentation, connector architecture, limitations
- Simon Willison analysis (2026-01-12): filesystem sandboxing mechanism (VZVirtualMachine), product positioning
- Microsoft Copilot Cowork partnership coverage (2026-03-09/10): enterprise vs consumer Cowork distinction, cloud vs local architecture
- Anthropic official blog post: folder access model, plan-execute-check loop, connector/skill architecture, Chrome integration

**Pre-build memo characterization:** "Supervised persistent agent. Desktop dispatch via Apple Virtualization Framework. VM-based, file-sandboxed. Visible to operator. Task-level autonomy, not campaign-level. No economic sovereignty. No HEARTBEAT equivalent."

### Headline Structural Characterization

Claude Cowork is a supervised persistent agent with the following structural commitments: folder-scoped filesystem access (user explicitly grants access to specific folders), VM-based sandboxing (Apple Virtualization Framework running a custom Linux root filesystem), plan-execute-check task loop (Claude formulates a plan, executes steps, checks its own work, asks for clarification), parallel task queuing (multiple tasks processed simultaneously), human-in-the-loop at significant actions (Claude asks before taking consequential actions), connector/skill extensibility (MCP connectors for external services, skills for document/presentation creation), and Chrome integration for browser automation.

Cowork is architecturally Claude Code wrapped in a non-terminal interface with a filesystem sandbox configured automatically. The underlying agent is the same (Claude Agent SDK). The key structural differences from Claude Code: non-technical user surface, folder-level rather than repo-level scoping, VM-based rather than direct filesystem sandboxing, and task-dispatch rather than code-authorship orientation.

The Microsoft Copilot Cowork partnership (March 2026) reveals a significant architectural fork: Anthropic's Cowork runs locally on the user's device with folder-level sandboxing; Microsoft's Copilot Cowork runs in the cloud within the M365 tenant with enterprise data graph access. Both use the same "agentic harness" but differ in deployment model, data access, and governance boundaries.

### Scaffold Readiness Assessment

**Verdict: MARGINAL — scaffold production is possible but would produce a thin artifact.** Evidence covers the architectural design, sandboxing mechanism, task execution model, and extensibility architecture. However: no structured decomposition of Cowork's primitives exists in the corpus; the surface is less than three months old with rapid iteration; and the Microsoft Copilot Cowork fork introduces a variant that may need its own treatment or a shared-architecture approach. A surface scaffold could be produced from web-sourced evidence, but it would be thinner than any SaaS or reasoning/chat scaffold and would carry higher staleness risk (high-ρ surface in active development).

### Gap: What additional evidence is needed

- Detailed connector/skill architecture documentation (what MCP connectors are available, what skills ship natively)
- Sandboxing boundary specification (what exactly can and cannot escape the VM)
- Task planning/execution loop detailed mechanics (is this identical to Claude Code's loop or adapted?)
- Chrome integration architecture and trust boundary
- Microsoft Copilot Cowork distinction: is this a separate surface or a deployment variant of the same surface?
- Operational experience documentation (the corpus has zero Cowork operational experience)
